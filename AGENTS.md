# AGENTS.md

## Leader Agent – 以 Spec 协同为主的开发模式

本项目优先采用 **Spec-Driven Development（规范驱动开发）** 进行沟通与协同，
Spec 是需求与实现之间的**唯一权威对齐文档**。在 Spec 确认后，Agent 必须严格按照 Spec 进行实现。

Spec 的结构和格式不做刚性统一约束，以「帮助双方讲清楚需求与约束」为唯一目标；
但无论采用何种形式，Agent 都应具备：

- 能够主动为每个模块/需求撰写清晰、可执行的 Spec；
- 在实现阶段严格参照已确认的 Spec 编码，如需偏离必须先更新 Spec 并取得确认。

***

# 一、开发流程（强制）

## Step 1：需求沟通阶段

当用户提出新需求或修改需求时，Agent 应该优先做「问清楚」而不是「先写模板」：

1. 主动提出澄清问题，包括但不限于：
   - 业务目标（为什么要做）
   - 功能范围（做到什么程度算完成）
   - 是否影响现有模块
   - 是否涉及数据库
   - API或页面是否变化
   - 边界情况与异常处理
2. 沟通形式可以灵活：
   - 可以通过多轮问答细化需求；
   - 可以结合现有页面 / 接口举例确认；
   - 必要时用一小段「文字草稿」替代完整 Spec 模板。

在需求明显复杂或影响范围较大时，更建议先沉淀 Spec 再实现；
对于任何涉及业务代码的修改（包括非常小且局部的变更），都必须遵守「先 Spec、后实现」：

- 不允许在同一轮对话或同一轮实现中，同时修改 Spec 与业务代码；
- 必须先单独更新或编写 Spec，并等待用户明确确认；
- 只有在 Spec 被确认之后，才可以开始编写或修改业务代码。

***

## Step 2：生成 Spec 文档（强制优先）

在进入**任何非纯文档/配置的小规模实现之前**，必须先生成或更新对应的 Spec 文档，并以此作为后续实现与验收的唯一依据：

```id="spec-path"
/specs/spec_<模块名>.md
```

Spec 的内容结构不强制统一，但至少应清晰说明（可按需要增减或合并），并遵守「每个 Spec 只描述本模块内容」的原则：

- 功能背景 / 目标（针对当前模块）
- 功能需求（当前模块的核心流程和主干场景）
- 非功能要求（如性能、安全、可维护性等，对当前模块的要求）
- 涉及模块与依赖关系（可以引用其他模块名称，但不在此定义其他模块的业务细节）
- API 变更（如有，针对当前模块暴露或依赖的接口）
- 数据库变更（如有，与当前模块直接相关）
- 验收标准或自测要点

无论采用何种段落结构，Spec 必须尽量准确反映当前预期的业务行为和技术约束，
避免「写一套、做一套」。

生成或更新后必须提示用户，并在**得到用户明确确认之前不得开始编码**：

**“请确认该 Spec，确认后我再开始编码。”**

***

## Step 3：确认与调整（必须等待）

在用户明确回复以下内容之一之前：

- 确认
- 通过
- 开始实现
- approved

在用户确认 Spec 之前，Agent 不得对业务逻辑进行实质修改；如确需先做小范围实验性改动（如验证第三方接口、原型验证等），必须在回复中说明目的与范围，并在用户确认 Spec 后再扩展实现范围。

***

# 二、实现阶段

确认后，Agent 可以并必须：

1. 仔细阅读并理解 Spec，将其视作实现的唯一来源（source of truth）；
2. 分析影响范围，评估对现有模块的影响；
3. 严格按照 Spec 约定的功能、接口和约束进行**最小范围增量修改**；
4. 保持现有架构不变，不在 Spec 之外随意扩展需求或改变行为。

在实现过程中，如发现 Spec 不完整或需要调整：

- 应先在回复中说明需调整的点，并停止进一步代码改动；
- 仅更新或补充 Spec 的相关内容，不同时修改业务代码；
- 经用户确认新的 Spec 后，再按更新后的 Spec 继续实现。

禁止：

- 重写整个模块
- 大规模重构
- 引入新框架

***

# 三、Spec 变更流程（严格）

当已有 Spec 被修改时：

Agent 必须执行：

## Step 1：变更分析

输出影响说明：

- 修改内容
- 影响模块
- 是否影响数据库
- 是否影响API
- 风险评估

然后提示：

**“请确认变更影响，确认后再执行修改。”**

***

## Step 2：等待确认

确认后，才允许进行相关代码调整；如在实现过程中需对 Spec 再次调整，应简要说明变化，
并与用户对齐后继续，以保证「Spec 与代码」始终同步。

***

# 四、增量修改原则

Agent 必须：

- 只修改必要文件
- 不影响无关功能
- 保留现有结构
- 避免全量重写

***

# 五、强制验证

每次修改后必须修改测试用例，并执行测试

如果失败：

- 自动修复
- 重新运行
- 直到全部通过

***

# 六、安全限制

禁止行为：

- 新增技术栈
- 修改数据库结构（除非Spec明确）
- 删除现有模块
- 随意调整项目结构
- 跳过测试

***

# 七、默认交互原则

Agent 默认行为：

- 优先通过对话问清楚需求与边界（可以多轮、可以举例）；
- 需求不简单时，用简洁 Spec 或关键要点记录共识，不强制固定格式，但内容必须足够支撑实现；
- 对复杂或有风险的改动：先提问 → 再 Spec → 等确认 → 再实现，且实现过程严格对齐 Spec；
- 发现 Spec 与现状不符或实现需要调整时，先更新 Spec、取得确认，再修改代码；
- 不再允许对非常小、影响局部的改动在实现过程中临时补写 Spec，所有业务相关变更一律遵守「先 Spec（含确认）、后实现」。
