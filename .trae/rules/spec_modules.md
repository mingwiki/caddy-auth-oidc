# Spec 模块内容范围规则

本项目中的每个 Spec 文件（`/specs/spec_<模块名>.md`）必须遵守「只描述本模块内容」的约束。

---

## 1. 内容范围

- 单个 Spec 只描述对应模块本身的：
  - 功能目标与业务范围；
  - 接口与数据模型；
  - 页面或交互行为（如有）；
  - 与该模块直接相关的非功能要求。
- 可以在「依赖关系」中引用其他模块的名称，但不得在当前模块 Spec 中定义其他模块的业务细节或接口行为。

---

## 2. 跨模块约束写法

- 涉及多个模块的通用约束（例如统一鉴权策略、基础设施约定等）应放在：
  - 专门的基础设施或架构类 Spec（如 `spec_infra` 等），或
  - 独立的跨模块 Spec 文档。
- 单个业务模块的 Spec 不应承担「全局规范」角色，避免跨模块耦合与描述重复。

---

## 3. 实践要求

- 在新增或更新 Spec 时，优先判断内容是否只影响当前模块：
  - 若只影响本模块：写入对应 `spec_<模块名>.md`；
  - 若同时影响多个模块：抽取到单独的跨模块 Spec，并在各模块 Spec 中以「引用」方式说明依赖。
